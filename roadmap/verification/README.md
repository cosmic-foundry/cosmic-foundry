# Meta-Level Roadmap

This plane is the roadmap for reproducibility, verification, validation,
provenance, and evidence. It answers **how Cosmic Foundry proves and
regenerates object-level claims**.

The companion [object-level roadmap](../implementation/README.md) answers
what platform and simulation capabilities the codebase is building.

## Framing

**Goal.** Build an evidence system that can describe, verify, validate,
and regenerate a physics engine state across the platform and application
repository family.

**Principle - executable evidence before scale.** Object-level claims
should be backed by executable meta-level evidence as early as possible:
specs, formulas, derivations, golden fixtures, validation manifests,
capsules, dry-run checks, and eventually regenerated evidence bundles.

**Principle - track selection per PR.** Each PR declares which track it
advances: object-level, meta-level, or both. The tracks can proceed in
parallel, but cross-track dependencies must be explicit.

**Principle - object-level claims stay out of the meta plane.** The
meta-level roadmap defines evidence machinery and acceptance workflow. It
does not decide scientific scope, numerical method choice, or platform
capability ordering except by recording explicit evidence dependencies.

## Verification Levels

Every capability claim sits at one of three levels. Each capability spec
must state which level it is claiming and what evidence supports that claim.

**Code verification.** Did we implement the equations correctly? The
standard method is the Method of Manufactured Solutions (MMS): choose a
smooth manufactured solution, derive the source term symbolically (using
SymPy), and verify that the solver converges to that solution at the stated
order as grid spacing h → 0. MMS is the primary external grounding method
for all discretization capabilities. Known limitation: smooth manufactured
solutions do not test discontinuities, stiff source terms, or boundary
condition implementation — these require separate targeted tests.

**Solution verification.** Is the discretization error acceptable for a
specific target problem? Method: Richardson extrapolation or the Grid
Convergence Index (GCI, Roache 1994) applied to a sequence of refined
grids on a physically meaningful problem. This is distinct from code
verification — a code can be correctly implemented but still produce
unacceptable solution error for a given problem at a given resolution.

**Model validation.** Does the physical model match reality? Method:
comparison to experimental data, observational data, or independently
published benchmark values — not to another simulation. This level belongs
in application repositories, which own domain physics and observational
data. The platform owns the schema and provenance machinery (M4); the
application owns the scientific content.

## The External Grounding Rule

A golden file generated by the engine is not external grounding. It is a
regression sentinel: it detects drift from a previously established
baseline, but it does not establish that the baseline was correct. A
map whose only tests compare against engine-generated data does not
satisfy M3.

External grounding means the expected answer comes from outside the
codebase: an analytical solution, a symbolic derivation committed
alongside the code, or a published benchmark value with a specific
citation. See ADR-0007 for the authoritative definition.

## Current Focus: M3 Convergence Coverage

What a physics map computes is stated in its `Map:` block (ADR-0016):
domain, codomain, operator, and approximation order p. That is the
implementation-track artifact. The verification-track question is whether
the stated p is correct: does the map actually converge at that order
against an externally grounded solution as grid spacing h → 0?

M3 is complete for a map when it has at least one convergence test — MMS
or analytical — that measures the actual exponent over a resolution
sequence and asserts it lies within an acceptable band of the stated p.
A single-resolution closeness check does not satisfy M3.

Once each implemented physics map has this coverage, M3b (the capsule
tooling) has something real to collect and dry-run against.

## Cross-Plane Interfaces

Meta-level work reads object-level state through explicit contracts:

- ADR indexes and architecture planes;
- `STATUS.md` current-position metadata;
- object-level roadmap files and epoch implementation plans;
- replication target plans and problem specs;
- formula register entries;
- derivation documents;
- manifest schemas and provenance sidecars;
- test selectors, convergence expectations, and validation thresholds;
- source maps and environment recipes.

Meta-level work writes evidence artifacts:

- reproducibility capsules;
- dry-run diagnostics;
- rendered independent-actor instructions;
- execution transcripts;
- evidence indexes;
- normalized capsule comparison reports.
